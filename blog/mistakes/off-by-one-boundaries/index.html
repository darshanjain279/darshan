The Boundary Condition I Have to Check Every Single Time

My mistake log's top category isn't some subtle algorithmic error. It's `<` instead of `<=`. Twice in one week.

CategoryMistakes
Published10 Apr 2026
Reading time5 min
dsaarraystwo-pointers

I keep a running log of every mistake I make on these practice problems, tagged by category and severity, mostly so I can stop lying to myself about which mistakes are random and which ones are patterns. The highest-severity recurring category, by a wide margin, is off-by-one errors. Not "I misunderstood the algorithm." Just: the loop ran one iteration too many, or one too few, because I trusted a boundary condition instead of actually checking it.

The setup

Two-pointer problems keep coming down to one line for me: the loop condition. while (l < r) versus while (l <= r). for i in range(len(nums)) versus range(len(nums) - 1). When I'm skimming my own code these look interchangeable, and I do skim them — by the time I've written the pointer-update logic in the middle of the loop, my brain's already decided the boundary is "obviously" whatever felt natural, and moved on without actually checking it. That's what burned me twice in the same week.

What went wrong

First was "Boats to Save People." Two pointers closing in from either end of a sorted array, pairing the lightest and heaviest remaining people into boats. I wrote the loop as while (l <= r). Looked harmless — equal counts as "still someone left," right? Except when l == r there's exactly one person left, who needs exactly one boat, and my condition let the loop run a full closing iteration for that case anyway, incrementing the boat counter for a step that should never have happened as a pair-closing move. I was double-counting the boat for the last remaining person. Traced it by hand against a 3-person input and watched my answer come out one boat too high, every single time, for any odd-length input. Even-length inputs never hit l == r, so they passed silently, and I'd nearly convinced myself the function was done before I happened to try an odd one.

Second was "Two Sum," a problem I'd have told you I was past making mistakes on, which is probably exactly why I made one. I wrote the brute-force check as for i in range(len(nums) - 1). Felt right because I was thinking about the inner loop needing i + 1 as a valid starting point, and some part of my brain smeared that "minus one" onto the outer loop too, for a symmetry that doesn't actually exist. The result: the last element of the array was never even considered as nums[i]. If the only valid pair involved the final index, my function returned nothing, silently, no error, just a wrong answer that looked exactly like "no solution exists."

The one that hasn't happened yet

There's another category sitting right next to off-by-one in the same log: not handling edge cases — empty input, a single-element array, an array where every value is identical. Right now that entry just says "none logged yet." Nothing's actually caught me on it. I'm putting it in this post anyway because it feels dishonest to only write about the stuff that's already blown up, when I'm already fairly sure this is next. Maybe I'm wrong and it never happens. I'd rather look a little silly for that than pretend I didn't see it coming.

The fix

The actual fix for both Boats and Two Sum was tiny — l < r instead of l <= r, range(len(nums)) instead of range(len(nums) - 1). Tiny is sort of the problem. A one-character diff doesn't jump out when I'm scanning my own code, because my eyes read what I meant to write, not what's actually on the screen. What I'm trying now is tracing two iterations by hand before I call a loop done — the very first one, with whatever the starting indices actually are, and the very last one, with the indices at whatever value makes the loop condition go false. For Two Sum that means actually asking out loud, "does i ever equal len(nums) - 1?" instead of glancing at the range() call and assuming it's fine. For the edge-case stuff, same idea before I write a single line: what does this do on empty input, one element, all-identical values. I've said "I'm going to start doing X" in this log before and not kept it up, so I'm not going to pretend this one's guaranteed to stick either.

The lesson

Both of these come down to the same thing, honestly — not thinking through the edges of a problem, whether the edge is a loop index or a weird input. The middle of the logic, the actual algorithm, is usually the part I get right, probably because it's the part I actually pay attention to. The boundary is the part I assume I already know, so I don't look at it closely. Going forward — supposedly — for any loop with two indices or a moving boundary, I'm going to write out what happens when the indices are equal, or adjacent, or sitting at the start or end of the array, before I write the loop body itself. I say "supposedly" because I've made myself promises like this before. The boundary keeps not feeling like it needs checking, which is apparently the whole reason it keeps getting me.
