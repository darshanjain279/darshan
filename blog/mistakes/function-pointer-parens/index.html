The Missing Parentheses That Turned My Callback Into Something Else

I wrote a function-pointer parameter, minus two parentheses. It compiled without a single complaint — because what I'd actually written was a completely different, equally legal declaration.

CategoryMistakes
Published1 Jul 2026
Reading time5 min
cfunction-pointers

I wrote a function signature that was supposed to take a callback, looked at it for a couple seconds, decided it was fine, and moved on. It compiled with zero warnings. Turns out that didn't mean much — what I'd actually typed wasn't a function pointer at all. It just happened to be legal C that looked close enough that I didn't catch it.

The setup

I was messing around with a small dispatcher — a function that takes two ints plus something telling it how to combine them, so I could pass in add or sub instead of writing five near-identical functions. I think this pattern shows up again later on, interrupt handlers or dispatch tables or something like that, not totally sure yet, but that's part of why I wanted to get comfortable with it now instead of later. I wrote:

int func(int a, int b, int *p(int, int));

In my head, p was a pointer to a function taking two ints and returning an int, and func would call p(a, b) with whatever got handed to it.

What went wrong

Turns out p isn't a pointer to a function. It's a function — takes two ints, returns int *. Not a pointer variable holding an address, an actual function whose return type happens to be a pointer. I didn't get this on my own, someone had to walk me through it: () right after an identifier apparently binds to that identifier before * does, so *p(int, int) reads as "p is a function of (int, int)," and the int * out front just describes what that function returns. I'm still a little fuzzy trying to explain it back to myself without looking at an example.

What got me is that none of this throws an error. It's legal C, just not the C I meant. A real typo stops you cold and tells you something's wrong. This one let me keep going, fully convinced I'd built a function-pointer parameter, right up until it would've actually mattered. I went back and stared at the line for a solid minute after it was pointed out, trying to see the wrong reading myself instead of just being told it, and it still took a second pass before it actually clicked instead of just sounding plausible.

The fix

int func(int a, int b, int (*p)(int, int));

Putting parens around *p apparently forces the grammar to group those first — p becomes the pointer, and then "function of (int, int) returning int" attaches to that pointer instead of floating off to some return-type slot. I've started trying to read these right to left, name first: p is a pointer, to a function taking (int, int), returning int. Whether that habit actually sticks past this week, I don't know yet.

A few days later there was a trace-through question testing basically the same thing from a different angle: given int *fp2(int, int); fp2 = add;, what is fp2? I got it right that time — function declaration, not a pointer variable — which meant the next line couldn't compile at all, since you can't assign to a function. That felt like a better sign than getting the first one right, honestly, like maybe the correct version actually replaced the wrong one in my head instead of just patching that one specific line.

The lesson

I think the rule is that function pointer syntax is always return_type (*name)(param_types), and the parens around *name are doing actual work, not just style — leave them out and you get a function declaration instead of a pointer. I'm going to try to actually check for this now, any time I write something mixing * and (): read it right to left starting at the name. Whether I'll remember to do that reliably or just get caught by this exact thing again in three weeks, no idea. Past experience says it's probably not a one-and-done fix. I've thought a rule was "locked in" before and then watched myself do the exact same thing a week later on a slightly different line, so I'm trying not to feel too confident about this one just because I can explain it right now.
